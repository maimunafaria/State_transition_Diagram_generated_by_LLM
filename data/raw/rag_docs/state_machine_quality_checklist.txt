Title: State Machine Quality Checklist

Purpose:
Use this checklist to validate a generated UML state machine before accepting it.

States:
- Keep state names short, concrete, and mutually distinct.
- Use verb phrase for transient states (e.g., Processing Payment).
- Use noun phrase for stable states (e.g., Active, Suspended).

Events:
- Label transitions with observable events when available.
- Use one event name per transition label.
- Keep event names lowercase snake_case or short phrase consistently.

Required transitions:
- Exactly one initial transition: [*] --> SomeState
- Every non-initial state should be reachable from the initial path.
- Terminal behavior should be explicit with either final state or sink policy.

Forbidden transitions:
- Do not use [*] --> [*].
- Do not add duplicate transitions with same source/event/target.
- Do not create disconnected state clusters unless requirement explicitly says so.

Terminal states:
- Final states should represent a completed or abandoned lifecycle.
- If cancellation exists, model it as an explicit terminal or resumable state.

PlantUML mini example:
@startuml
[*] --> Draft
Draft --> Submitted : submit
Submitted --> Approved : approve
Submitted --> Rejected : reject
Approved --> [*]
Rejected --> [*]
@enduml
