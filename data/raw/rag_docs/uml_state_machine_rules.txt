UML State Machine Rules

1. A state machine contains a single initial state represented as [*].
2. A state represents a condition, phase, or mode in the lifecycle of an object.
3. Transitions describe movement from one state to another, triggered by events.
4. Basic syntax:
     [*] --> StateName
     StateA --> StateB : eventName

5. Final states are optional and represented as:
     StateX --> [*]

6. Events are verbs or actions that cause state change.
   Examples: open, close, submit, confirm, authenticate, reset.

7. Common modeling errors:
   - Missing initial state
   - Undefined states referenced in transitions
   - Orphan states with no transitions
   - Missing or ambiguous events
   - Incomplete transition coverage

8. A good state machine:
   - Is deterministic
   - Avoids redundant states
   - Follows logical domain behavior
   - Has clear naming conventions
