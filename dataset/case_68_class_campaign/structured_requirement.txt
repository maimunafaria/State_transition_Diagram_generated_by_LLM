Class Campaign — Polished Requirement Specification

Overview
The transition from the initial pseudostate to the Commissioned state has been labelled only with an activity-expression that comprises the operations assignManager and assignStaff. Execution of these operations ensures that when a campaign is created a manager and at least one member of staff are assigned to it.4 The operations are triggered by the event that creates a Campaign object. The transition from the Completed state to the Paid state has a guard condition that only allows the transition to fire if total amount due (paymentDue) for the Campaign has been completely paid (note that this guard condition allows a Campaign to enter the Paid state when the client overpays).

Operational Context
Primary actors include user.

Functional Requirements
1. The system shall support the transition from the initial pseudostate to the Commissioned state has been labelled only with an activity-expression that comprises the operations assignManager and assignStaff.
2. The system shall ensure that execution of these operations ensures that when a campaign is created a manager and at least one member of staff are assigned to it.4 The operations are triggered by the event that creates a Campaign object.
3. The system shall support the transition from the Completed state to the Paid state has a guard condition that only allows the transition to fire if total amount due (paymentDue) for the Campaign has been completely paid (note that this guard condition allows a Campaign to enter the Paid state when the client overpays).
4. The system shall support the reflexive transition from the Completed state models any payment event that does not reduce the amount due to zero or less.
5. The system shall ensure that only one of the two transitions from the Completed state (one of which is recursive) can be triggered by the paymentReceived event since the guard conditions are mutually exclusive.
6. The system shall support it would be bad practice to construct a state machine where one event can trigger two different transitions from the same state.
7. The system shall ensure that A state machine is only unambiguous when all the transitions from each state are mutually exclusive.
8. The system shall ensure that f the user requirements were to change, so that an overpayment is now to result in the automatic generation of a refund, the state machine could be changed.
9. The system shall ensure that since the action that results from an overpayment is different from the action that results from a payment that reduces paymentDue to zero, a new transition is needed from the Completed state to the Paid state.
10. The system shall ensure that it is important to appreciate that the state machines in Figs 11.6 and 11.7 are not equivalent to each other, but capture different versions of the users’ requirements.
