Java — Polished Requirement Specification

Overview
consider the state diagram for a part of a Java compiler. The input to the compiler is a text file, which can be thought of as a long string of characters. The compiler reads characters one at a time and from them determines the structure of the program.
To help you visualize all the states of this object and how it changes state, you can use a UML state diagram as shown in Figure A1.13.

Operational Context
Primary actors are not explicitly stated in the source text.

Functional Requirements
1. The system shall ensure that the input to the compiler is a text file, which can be thought of as a long string of characters.
2. The system shall support suppose that the compiler delegates to a WhiteSpaceAndCommentEliminator the job of advancing over white-space characters and characters in comments.
3. The system shall ensure that that is, this object’s job is to read input characters until all white-space and comment characters have been read, at which point it returns control to the compiler to read and process non-white-space and noncomment characters.
4. The system shall support the object can check for white space by testing the next character against “ ”, “\t”, “\n”, and “\r”.
5. The system shall support make this determination, WhiteSpaceAndCommentEliminator needs to make a note of the fact that it saw a “/” and then move on to the next character.
6. The system shall ensure that if the character following the “/” is another “/” or an “*”, then WhiteSpaceAndCommentEliminator knows that it is now reading a comment and can advance to the end of the comment without processing or saving any characters.
7. The system shall ensure that in summary, as WhiteSpaceAndCommentEliminator reads in characters, it needs to keep track of several things, including whether the current character is white space, whether the previous character it read was a “/”, whether it is currently reading characters in a comment, whether it has reached the end of comment, and so forth.
8. The system shall support these all correspond to different states of the WhiteSpaceAndCommentEliminator object.
9. The system shall support in each of these states, WhiteSpaceAndCommentEliminator behaves differently with regard to the next character read in.
10. The system shall support arrows from one state to another state indicate transitions or changes in the state of the object.
11. The system shall ensure that each transition is labeled with a trigger event, a slash (/), and an activity.
12. The system shall ensure that all parts of the transition labels are optional in state diagrams.
13. The system shall ensure that if the object is in one state and the trigger event for one of its transitions occurs, then that transition’s activity is performed and the object takes on the new state indicated by the transition.
14. The system shall ensure that if the character after the “/” is another “/”, then the object advances to the line comment state and stays there until it reads.
